# 项目协作说明

## 用户个人协作偏好（跨项目通用）

- 沟通语言默认使用中文，回答以直接、清楚、可执行为主，不要过度空话。
- 用户更喜欢“先做再汇报”的合作方式：
  - 小改动直接落地
  - 大改动、算法重构、布局重排前先给一个简短方案
- 非必要不要频繁追问；若存在明显分支选择，先给 2 到 3 个可选方案，再建议默认方案。
- 用户对“回归问题”非常敏感：
  - 已经做对的布局、文案、交互不要轻易重写
  - 修改新功能时，优先局部补丁式调整，避免牵连整个页面
  - 一旦出现原本显示正常的模块消失、错位、重叠、穿模，必须优先修复
- 用户偏好“保留原格局、升级质感”的设计方式：
  - 不希望频繁推翻原页面结构
  - 更希望在原有信息架构基础上做高级感、美化和交互增强
- 前端视觉偏好明确：
  - 喜欢更高级、更精致、更有品牌感的界面
  - 交互风格偏好参考苹果、小米、亚朵等产品的顺滑、柔和、细腻反馈
  - 不喜欢廉价感、过素、像低配后台模板的视觉效果
- 动效偏好：
  - 切换、跳转、定位、展开、悬浮都要“柔和、顺滑、自然”
  - 不要突然弹出、突兀位移、像卡顿一样的硬切
  - 同一页面内的板块跳转要尽量有空间连续性
- 表单/滚轮/选择器偏好：
  - 优先做成点击当前输入框后，就地弹出选择层
  - 弹层应覆盖在上方，不应被下方内容遮挡
  - 选完后自动收回
  - 鼠标悬浮、聚焦、选中时需要有明显但不廉价的反馈
- 页面布局偏好：
  - 信息要完整显示，不能重叠、不能裁切、不能因为一处错误导致整块空白
  - 要充分利用空白区域，但不要堆得臃肿
  - 概览页、首页类页面需要重点保证首屏清晰、重点明确
  - 手机端如果现有桌面布局迁移效果差，可以接受“移动端独立设计”，不要求与桌面端同构
- 数据与内容偏好：
  - 推荐内容尽量贴近日常中国家庭饮食习惯
  - 健身、减脂、维持、外食等模式要真实有差异，不要只是名字不同
  - 食物、餐厅、运动数据尽量丰富，并优先采用权威来源或固定规则整理
- 图片与视觉素材偏好：
  - 喜欢“拟真手绘、本地化、可一眼识别”的食物图
  - 餐厅品牌类素材可以保持品牌宣传图风格
  - 不喜欢所有食物共用一张图，或者图像和食物不匹配
- 验证要求较高：
  - 前端改动后，不能只做代码层检查，最好实际打开页面验证
  - 重点检查登录、注册、概览、饮食记录、滚轮弹层、定位跳转、卡片显示
  - 需要主动检查全屏和常规窗口下是否有错位、遮挡、留白异常
- Git/交付偏好：
  - 用户希望最终结果可直接使用，少走弯路
  - 如果涉及 Git、远端、发布、Docker，尽量一次整理成可持续使用的方式
  - 完成后最好直接告诉用户关键命令、访问地址或推送结果

## 本轮已确认的移动端重构决议

- 当前项目允许“桌面端保持现状，移动端独立重写展示层”的方案：
  - 桌面端页面结构、布局和现有效果默认不动
  - 移动端可采用完全不同的信息架构和视觉层级
  - 前后端接口、数据逻辑、业务规则继续复用，不重写后端
- 移动端设计方向已确认：
  - 视觉参考偏苹果系高级简约感
  - 色彩方向已从早期暖白方案调整为更冷静的雾蓝绿 / 冰感浅灰体系
  - 主观感受要求：清新、高级、克制，不要米白、奶油黄、低配后台模板感
  - 少玻璃感、少厚重阴影、少复杂装饰，强调克制、留白、卡片秩序和原生感
- 移动端交互方向已确认：
  - 不再强行复用桌面端“首页大 Hero + 顶部 7 功能入口”的布局
  - 改为更接近 App 的导航方式，优先采用底部 4 主导航
  - 记录类、趋势类、设置类功能在移动端内部再做二级分组，不要求和桌面端一一对应
- 移动端性能要求已提升为硬约束：
  - 默认减少重 blur、重阴影、大面积 backdrop-filter、复杂 sticky、跟手光效、过多动画
  - 优先保证点击响应、滚动流畅、切换稳定

## 本轮新增移动端细化规则

- 移动端首页已明确不再保留“快捷记录”首屏入口：
  - 用户认为其与“记录”主导航重复
  - 首屏优先留给更高价值的信息密度
- 移动端首屏信息必须避免重复：
  - 同一个指标不要同时出现在“今日状态”“核心指标”“目标完成度”多个区块
  - 若有重复，优先保留更能指导行动的位置，删掉展示型重复项
- 移动端首页信息排序应优先考虑：
  - 当前体重 / 目标体重
  - 到达日期 / 坚持天数
  - 运动消耗 / 目标总热量
  - 再考虑其它解释性信息
- 移动端记录页要求：
  - 饮食 / 运动 / 睡眠 / 体重四个分栏必须稳定切换
  - 不允许因为模块重挂载顺序导致某个分栏空白或消失
  - 运动记录区可以继续压缩高度，但不能牺牲表单可点性
- 移动端时间默认值要求：
  - 饮食 / 运动 / 体重记录时间默认值必须使用“当前时刻”
  - 不能停留在登录或进入系统时的固定分钟
  - 如果用户手动改过时间，后续默认刷新不能强行覆盖用户输入
- 移动端趋势页要求：
  - 重点压缩纵向高度，减少“长卡片 + 长滚动”
  - 体重曲线统计块优先做成 `2 + 1` 结构压缩高度
  - 报告中心顶部冗余说明、插图可在移动端隐藏
  - 身体评估在移动端趋势页只保留“评估结果”，不保留完整输入表单
- 移动端我的页要求：
  - 顶部“账户信息 + 退出登录”优先做成单行
  - 设置模块中的插图在移动端可隐藏
  - 整体以压缩上下距离、提升信息密度为主
- 移动端视觉微调原则：
  - 字号、padding、gap 可以持续压缩，但要保持点击区域可用
  - 优先压缩“展示性留白”，不要压坏输入框、按钮、滚轮、图表可读性
- 工作方式要求：
  - 用户已经明确不希望继续“改一点看一点”的无效往返
  - 做移动端改动前应先分析重复项、层级、挂载逻辑和密度问题
  - 改完要先做自检，再让用户同步看效果

## 本轮新增移动端精修约束（2026-06-24）

- 当前阶段仍然只精修移动端：
  - 桌面端默认完全不动
  - 如需改样式，优先放在移动端断点或移动端壳层作用域内
- 移动端首页顶部“健康管理 / 今日首页”这类抬头不跟随下滑：
  - 用户明确不希望它在滚动时一直跟下来
  - 首页上方展示性抬头可以在继续下滑时自然消失
- 移动端记录页的 sticky 规则已明确：
  - 记录页顶部大标题区不需要 sticky
  - 只有“饮食 / 运动 / 睡眠 / 体重”四个切换 tab 需要吸顶并跟随
- 移动端记录页的定位规则已明确：
  - 点击底部“记录”主导航时，应默认回到当前记录卡片顶部
  - 饮食记录提交成功后，应重新对齐到“饮食记录”卡片顶部
  - 这类定位优先对齐 `.module-head`，而不是停在搜索中段或结果中段
- 移动端输入法与底部导航的关系已明确：
  - 底部“首页 / 记录 / 趋势 / 我的”只在输入法真正弹出时隐藏
  - 不能因为只是点中了输入框就提前消失
  - 输入法收回后，底部导航要立即恢复显示
  - 这类判断优先依据真实 viewport 高度变化，而不是单纯 focus 状态
- 移动端饮食记录成功反馈已明确：
  - 除表单下方的 `food-log-result` 外，还需要一个屏幕中下方的轻提示 toast
  - 文案格式可采用“食物名 + 克数 + 已记录成功”
  - toast 停留时间应短，约 1 秒左右自动消失，不要打断后续录入
- 移动端饮食连续录入体验已明确：
  - 饮食记录提交成功后，应清空搜索框关键词
  - 同时清空搜索结果和当前选中食物，方便直接录入下一条
  - 不能让用户每次成功后还要手动删关键词再继续搜
- 移动端饮食搜索结果卡片精修规则已明确：
  - 当前方向是继续压缩卡片纵向高度，但不牺牲识别度和可点性
  - “我是 xxx” 这类装饰性 badge 在移动端不需要显示
  - 优先压缩图片上下留白、图片区背景空白、卡片内展示性间距
  - 文字主信息、按钮可点面积、图片识别度优先保留
- 移动端趋势页 BMI 展示规则已明确：
  - BMI marker 不能在高值时越出右边界
  - 需要保证不同手机宽度下 marker 和数值 label 都留在图表容器内
- 移动端成功优先级规则已明确：
  - 先保证“能提交、能记录、能回位、能继续下一条”
  - 再处理质感、美化和卡片高度压缩
  - 如果视觉优化影响到提交、滚动、定位、键盘或点击，必须回退到稳定方案

## 本轮新增移动端数值交互规范（2026-06-28）

- 当前阶段已确认：移动端数值输入相关问题不能再继续靠零散补丁维持。
  - 如果某一整套交互已经表现出“廉价感、bug 感、不稳定感”，优先整体重构，不继续叠补丁
  - 本轮典型对象就是移动端数值选择 / 滚轮 / 数字输入体系
- 移动端数值字段的统一原则已确认：
  - 注册页与进入系统后的移动端表单，不允许继续混用“原生 number 输入”和“自定义滚轮弹层”两套体验
  - 后续方向是统一为一套移动端自定义数值选择器体系
  - 桌面端默认不受影响，仍以现有行为为准
- 移动端数值选择器体验目标已确认：
  - 交互质感参考苹果系原生选择器，强调顺滑、稳定、克制、自然
  - 不允许出现明显乱蹦、二次吸附、滚动停不稳、回弹突兀、误触即弹出的表现
  - 不允许出现视觉上像故障、像补丁、像低配网页控件的状态
- 移动端数值字段与页面滚动的关系已确认：
  - 用户手指的首要意图如果是纵向滑页，不应因为起手碰到数值区域就直接弹出滚轮
  - 只有明确 tap / 短按命中数值字段时，才打开选择器
  - 页面纵向滚动优先级高于数值选择器唤起
- 移动端数值字段与输入法的关系已确认：
  - 目标是尽量避免原生数字键盘、原生步进器、原生 number spinner 破坏体验
  - 进入系统后的移动端数值字段，也要逐步并入统一弹层选择器，而不是继续保留原生数字输入感
- 数值默认值策略已进一步确认：
  - 默认值必须符合常识和前置选择逻辑，不能再出现明显不合理的起始值
  - 例如注册流程中，性别联动的默认体重应直观可感，不应回退到极低异常值
- 工作方式要求已补充：
  - 这一类交互改动，默认要先分析“是否是多套系统混用”再动手
  - 改完不能只看代码层，至少要做移动端实际交互验证
  - 若局部修复无法消除 bug 感，应直接进入同类控件统一重构
  - 重构过程中仍然遵守“桌面端不动、移动端保留原格局、只升级质感”的总原则

## 项目概览

- 项目名称：轻量级健康管理 Web 系统

  - 部署：Docker Compose

- 后端接口已包含：

- 当前移动端已做过若干轮局部补丁，但已确认仅靠桌面端缩放补丁无法满足长期体验目标
- 当前后续方向不是继续小修移动端，而是逐步替换为移动端独立 UI 壳层
- 当前移动端独立壳层已经落地：
  - 已有独立首页、记录、趋势、我的四主屏
  - 已有移动端底部导航和记录页二级切换
  - 已通过 DOM 重挂载复用原有饮食 / 运动 / 睡眠 / 体重 / 报告 / 设置逻辑
- 当前移动端登录 / 注册页已被用户明确认可为“完美”
- 当前移动端仍处于“信息密度精修阶段”：
  - 重点是去重、压高度、调顺序、修单屏可视效率
- 当前 Docker 同步习惯：
  - 开发期通常直接 `docker cp` 到现有容器，不一定每次 `docker compose up -d --build`
- 当前体型功能定位：

  - 经典热量算法仍为主算法
  - `体型参考版` 仅作为预留选项，不应在没有可靠依据前强行改变核心公式

## 本轮新增协作与回归规范（2026-06-29）

- 当前阶段已确认：用户希望减少“反复确认后再继续”的来回沟通。
  - 能直接落地并可自行验证的事项，默认先做完再汇报
  - 尤其是食物图片纠错、前端细节修正、容器同步、回归测试、Git 推送，不要拆成很多轮重复确认
- 当前阶段已确认：食物图片问题要优先保证“语义正确”，再追求质感升级。
  - 图像与食物不匹配，比使用通用图更严重
  - 对酒类、玉米、烤玉米这类用户已明确指出的问题，必须优先彻底修正
  - 做食物图修正时，优先检查是否被宽泛分类规则、短关键词、通用映射误命中
- 食物图片映射新增硬规则：
  - 具体食物命名映射优先级必须高于宽泛的分类写实图
  - 更长、更具体的关键词必须优先于更短关键词，避免 `烤玉米` 命中 `玉米`、`玉米油` 命中 `玉米` 这类串图
  - 若同一类食物已有独立素材，优先使用独立素材，不要退回到宽泛写实图或错误通用图
- 食物图片改动后的验证要求已提高：
  - 不能只靠静态代码判断，必须同步到前端容器并在浏览器里实际抽检
  - 至少验证用户点名问题项，以及若干高风险相邻项
  - 例如：`玉米`、`烤玉米`、`玉米油`、典型酒类、典型水果、典型主食
- 当前阶段已确认：整程序回归必须测试“真实提交链路”，不能只看页面显示正常。
  - 登录、注册、饮食记录、运动记录、睡眠记录、体重记录都要至少走一遍真实提交
  - 不仅要看前端提示成功，还要确认后端接口确实被调用、数据库确实写入、首页/报告确实同步
  - 若前端显示成功但数据库无记录，要按真实 bug 处理，不能视为通过
- 前端表单新增验证规范：
  - 所有自动计算后会自动回填的数值字段，前端输入约束必须兼容系统生成值
  - 不允许出现“页面里展示了系统自动值，但表单原生校验拦住提交”的情况
  - 本轮已确认案例：运动消耗自动估算会出现两位小数，因此相关 `number` 输入的 `step` 不能低于 `0.01`
- 容器开发规范补充：
  - 前端改动后，若当前开发习惯是 `docker cp` 同步，则同步后必须重启 `health_frontend` 再验证
  - 回归时如发现行为和代码不一致，优先排查是否未同步容器，而不是先怀疑业务逻辑
- Git 交付规范补充：
  - 修完并验证通过后，默认推送到 `origin` 与 `github`
  - 若本轮只产生单点修复，也应单独形成明确提交，便于后续追踪问题来源
