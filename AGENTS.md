# 项目协作说明

## 用户个人协作偏好（跨项目通用）

- 沟通语言默认使用中文，回答以直接、清楚、可执行为主，不要过度空话。
- 用户更喜欢“先做再汇报”的合作方式：
  - 小改动直接落地
  - 大改动、算法重构、布局重排前先给一个简短方案
- 非必要不要频繁追问；若存在明显分支选择，先给 2 到 3 个可选方案，再建议默认方案。
- 用户对“回归问题”非常敏感：
  - 已经做对的布局、文案、交互不要轻易重写
  - 修改新功能时，优先局部补丁式调整，避免牵连整个页面
  - 一旦出现原本显示正常的模块消失、错位、重叠、穿模，必须优先修复
- 用户偏好“保留原格局、升级质感”的设计方式：
  - 不希望频繁推翻原页面结构
  - 更希望在原有信息架构基础上做高级感、美化和交互增强
- 前端视觉偏好明确：
  - 喜欢更高级、更精致、更有品牌感的界面
  - 交互风格偏好参考苹果、小米、亚朵等产品的顺滑、柔和、细腻反馈
  - 不喜欢廉价感、过素、像低配后台模板的视觉效果
- 动效偏好：
  - 切换、跳转、定位、展开、悬浮都要“柔和、顺滑、自然”
  - 不要突然弹出、突兀位移、像卡顿一样的硬切
  - 同一页面内的板块跳转要尽量有空间连续性
- 表单/滚轮/选择器偏好：
  - 优先做成点击当前输入框后，就地弹出选择层
  - 弹层应覆盖在上方，不应被下方内容遮挡
  - 选完后自动收回
  - 鼠标悬浮、聚焦、选中时需要有明显但不廉价的反馈
- 页面布局偏好：
  - 信息要完整显示，不能重叠、不能裁切、不能因为一处错误导致整块空白
  - 要充分利用空白区域，但不要堆得臃肿
  - 概览页、首页类页面需要重点保证首屏清晰、重点明确
  - 手机端如果现有桌面布局迁移效果差，可以接受“移动端独立设计”，不要求与桌面端同构
- 数据与内容偏好：
  - 推荐内容尽量贴近日常中国家庭饮食习惯
  - 健身、减脂、维持、外食等模式要真实有差异，不要只是名字不同
  - 食物、餐厅、运动数据尽量丰富，并优先采用权威来源或固定规则整理
- 图片与视觉素材偏好：
  - 喜欢“拟真手绘、本地化、可一眼识别”的食物图
  - 餐厅品牌类素材可以保持品牌宣传图风格
  - 不喜欢所有食物共用一张图，或者图像和食物不匹配
- 验证要求较高：
  - 前端改动后，不能只做代码层检查，最好实际打开页面验证
  - 重点检查登录、注册、概览、饮食记录、滚轮弹层、定位跳转、卡片显示
  - 需要主动检查全屏和常规窗口下是否有错位、遮挡、留白异常
- Git/交付偏好：
  - 用户希望最终结果可直接使用，少走弯路
  - 如果涉及 Git、远端、发布、Docker，尽量一次整理成可持续使用的方式
  - 完成后最好直接告诉用户关键命令、访问地址或推送结果

## 本轮已确认的移动端重构决议

- 当前项目允许“桌面端保持现状，移动端独立重写展示层”的方案：
  - 桌面端页面结构、布局和现有效果默认不动
  - 移动端可采用完全不同的信息架构和视觉层级
  - 前后端接口、数据逻辑、业务规则继续复用，不重写后端
- 移动端设计方向已确认：
  - 视觉参考偏苹果系高级简约感
  - 色彩方向已从早期暖白方案调整为更冷静的雾蓝绿 / 冰感浅灰体系
  - 主观感受要求：清新、高级、克制，不要米白、奶油黄、低配后台模板感
  - 少玻璃感、少厚重阴影、少复杂装饰，强调克制、留白、卡片秩序和原生感
- 移动端交互方向已确认：
  - 不再强行复用桌面端“首页大 Hero + 顶部 7 功能入口”的布局
  - 改为更接近 App 的导航方式，优先采用底部 4 主导航
  - 记录类、趋势类、设置类功能在移动端内部再做二级分组，不要求和桌面端一一对应
- 移动端性能要求已提升为硬约束：
  - 默认减少重 blur、重阴影、大面积 backdrop-filter、复杂 sticky、跟手光效、过多动画
  - 优先保证点击响应、滚动流畅、切换稳定

## 当前确认的移动端方案

- 推荐移动端底部主导航：
  - 首页
  - 记录
  - 趋势
  - 我的
- 推荐移动端页面结构：
  - 首页：今日状态、核心指标、目标完成度、今日建议
  - 记录：饮食 / 运动 / 睡眠 / 体重
  - 趋势：体重趋势、日报 / 周报、评估结果
  - 我的：账户信息、个人资料、目标设置、推荐偏好、常备食材、退出登录
- 登录 / 注册页：
  - 移动端可与桌面端完全不同布局
  - 整体以垂直居中、单卡片、原生感输入区为主
  - 注册流程允许按信息分组组织，避免一屏过长

## 本轮新增移动端细化规则

- 移动端首页已明确不再保留“快捷记录”首屏入口：
  - 用户认为其与“记录”主导航重复
  - 首屏优先留给更高价值的信息密度
- 移动端首屏信息必须避免重复：
  - 同一个指标不要同时出现在“今日状态”“核心指标”“目标完成度”多个区块
  - 若有重复，优先保留更能指导行动的位置，删掉展示型重复项
- 移动端首页信息排序应优先考虑：
  - 当前体重 / 目标体重
  - 到达日期 / 坚持天数
  - 运动消耗 / 目标总热量
  - 再考虑其它解释性信息
- 移动端记录页要求：
  - 饮食 / 运动 / 睡眠 / 体重四个分栏必须稳定切换
  - 不允许因为模块重挂载顺序导致某个分栏空白或消失
  - 运动记录区可以继续压缩高度，但不能牺牲表单可点性
- 移动端时间默认值要求：
  - 饮食 / 运动 / 体重记录时间默认值必须使用“当前时刻”
  - 不能停留在登录或进入系统时的固定分钟
  - 如果用户手动改过时间，后续默认刷新不能强行覆盖用户输入
- 移动端趋势页要求：
  - 重点压缩纵向高度，减少“长卡片 + 长滚动”
  - 体重曲线统计块优先做成 `2 + 1` 结构压缩高度
  - 报告中心顶部冗余说明、插图可在移动端隐藏
  - 身体评估在移动端趋势页只保留“评估结果”，不保留完整输入表单
- 移动端我的页要求：
  - 顶部“账户信息 + 退出登录”优先做成单行
  - 设置模块中的插图在移动端可隐藏
  - 整体以压缩上下距离、提升信息密度为主
- 移动端视觉微调原则：
  - 字号、padding、gap 可以持续压缩，但要保持点击区域可用
  - 优先压缩“展示性留白”，不要压坏输入框、按钮、滚轮、图表可读性
- 工作方式要求：
  - 用户已经明确不希望继续“改一点看一点”的无效往返
  - 做移动端改动前应先分析重复项、层级、挂载逻辑和密度问题
  - 改完要先做自检，再让用户同步看效果

## 项目概览

- 项目名称：轻量级健康管理 Web 系统
- 目标：基于本地数据库、固定公式、规则引擎实现饮食、运动、睡眠、体重记录与分析
- 禁止事项：不接入 AI 模型、不接入生成式推荐、不依赖外部智能接口
- 当前技术栈：
  - 后端：FastAPI + SQLAlchemy + Pydantic
  - 数据库：PostgreSQL
  - 前端：原生 HTML / CSS / JavaScript
  - 部署：Docker Compose

## 修改代码后必须执行的验证步骤

- 修改后端 Python 代码后必须执行：
  - `python3 -m py_compile backend/app/*.py`
- 修改前端 JavaScript 后必须执行：
  - `node --check frontend/app.js`
- 修改 `backend/app/foods.json`、`models.py`、`schemas.py`、`services.py`、`main.py` 任一文件后：
  - 必须重新同步后端容器并重启 `health_backend`
- 修改 `frontend/index.html`、`frontend/app.js`、`frontend/styles.css` 或 `frontend/assets/` 后：
  - 必须重新同步前端容器并重启 `health_frontend`
- 如果改动影响容器内运行结果，必须同步到运行中的容器或重建：
  - 前端快速同步：
    - `docker cp frontend/. health_frontend:/usr/share/nginx/html/`
  - 后端快速同步：
    - `docker cp backend/app/. health_backend:/app/app/`
  - 同步后建议重启对应容器：
    - `docker restart health_backend health_frontend`
- 至少验证以下接口或页面之一：
  - `curl -s http://localhost:8000/health`
  - 刷新 `http://localhost:8080/#/overview`
- 如果首页数据区显示不出来，优先检查：
  - `docker logs --tail 200 health_backend`
  - 重点确认 `/dashboard/summary` 是否报 500

## 代码风格约定

- 通用：
  - 默认使用 ASCII；除中文文案或已有中文文件外，不主动引入无必要的特殊字符
  - 尽量保持函数职责单一，命名直接清晰
  - 避免引入重量级新依赖，优先复用现有结构
- 后端：
  - 逻辑集中在 `services.py`
  - 路由集中在 `main.py`
  - 数据结构在 `schemas.py`
  - 模型变更时同步更新 `models.py` 和迁移逻辑
  - `DashboardSummaryResponse.targets` 当前是 `dict[str, float]`
  - 不要往 `targets` 中写入字符串、日期字符串或复杂对象，除非先同步改 schema
- 前端：
  - 继续维持单文件原生 JS 结构，不要突然切框架
  - 新增交互优先复用已有状态对象 `state`
  - 面板切换统一走 `showPanel` / `navigateToPanel`
  - 选择器/滚轮类交互优先复用已有 `picker` 风格
  - 概览页布局已多次被用户微调，默认先做局部修改，避免整体重写 `hero` 区
  - 例外：移动端如果已经确认采用独立方案，可在小屏断点下单独重写结构和视觉，不必强行与桌面端同构
  - 移动端新增导航或壳层时，优先复用现有数据加载与提交逻辑，不改接口契约
  - 移动端改版时要把“首屏可达性、单手操作、底部导航可持续切换”放在视觉之前
