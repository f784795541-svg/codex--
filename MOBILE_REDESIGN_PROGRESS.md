# 移动端独立重构进度

## 背景

- 现有桌面端布局在电脑浏览器中基本满足需求。
- 现有移动端体验已确认不适合继续用“桌面端缩放 + 局部补丁”的方式维护。
- 已决定：桌面端保持现状，移动端采用独立布局和独立视觉层级，但继续复用当前后端接口和前端业务逻辑。

## 已确认的决策

- 方向：重写移动端展示层，不重写后端。
- 桌面端：保持不动，避免引入回归。
- 视觉：参考苹果系高级简约感，当前已转为偏冷静的雾蓝绿 / 冰感浅灰配色，不再使用米白主调。
- 交互：更接近原生 App，弱化网页式大段首屏。
- 导航：采用底部 4 主导航方案。
  - 首页
  - 记录
  - 趋势
  - 我的
- 性能：移动端默认进入轻量模式，减少 blur、厚阴影、复杂动画和不必要的滚动跳转。
- 数据逻辑：继续复用桌面端现有接口、表单提交、图表渲染和后端规则，不改接口契约。

  - 移动端多轮密度压缩、定位回位、toast 反馈、搜索清空、BMI 越界修正等关键问题已经处理过一轮
  - 数值滚轮相关问题已经不再被视为单点小 bug，而是被明确识别为“同类交互体系不统一”带来的系统性问题
- 这阶段已经定了什么：
  - 桌面端不动，移动端继续独立精修
  - 数值输入这条线不再继续靠零散补丁修补，而是转向统一交互体系
  - 注册页数值选择器和系统内其它数值字段，后续要收敛到同一套移动端弹层选择器体验
  - 目标质感是苹果系原生感：顺滑、稳定、自然，不出现乱蹦、误触、廉价原生数字框感
  - 页面滚动优先级高于数值字段唤起，不能因为手指起手碰到字段就误开滚轮
- 这阶段已经确认的判断：
  - 当前最核心的问题不是单个样式点，而是“注册页一套、系统内一套、原生输入又一套”的混用状态
  - 只要移动端数值字段继续混用原生 `number` 和自定义 picker，体验就很难真正高级，也容易反复出现 bug 感
  - 因此后续应优先做移动端数值控件统一，而不是继续在旧行为上叠小修小补
- 现在还差什么：
  - 还没有完成“系统内全部移动端数值字段”向统一弹层选择器的彻底收口
  - 还需要把注册页之外的年龄、身高、体重、目标体重、运动时长、目标睡眠等字段逐步并入同一套移动端交互
  - 还需要继续做移动端真实点击与滚动验证，确认不会误弹、不会跳值、不会再唤起廉价原生数字输入体验
  - 等这一轮统一完成后，再继续做更细的高级感微动效和视觉精修，避免在不稳定控件上叠加装饰
  - 这也意味着当前移动端仍处于“结构已定，关键交互体系继续重构”的阶段，而不是完全收尾
  - 下一步默认优先级：先统一移动端数值选择体系，再做剩余视觉精修和真机回路检查
